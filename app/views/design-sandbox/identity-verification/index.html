Success page pattern

Whenever information is successfully added or changed, the user is presented with the appropriate success notification banner.

  • Back channel logout URI success
  • Claims success
  • Client secret hash success
  • ID token signing algorithm success
  • Public key success
  • Sector identifier URI success

List of screens with different scenarios

  • Blank version with IDV disabled

    • This view has minimal information inputted.

  • IDV enabled with claims and multiple contacts

    • This view has multiple contacts and redirect URIs entered and because the authentication method is 'private_key_jwt', public key is visible.

  • Authentication method: client_secret_post

    • A view where the authentication method is 'private_key_jwt' so client secret hash is visible.

Logic

Authentication method

  • When 'client_secret_post' is enabled, 'client secret hash' is shown.
  • When 'private_key_jwt' is enabled, 'public key' is shown.

Identity verification enabled

  • When Idenity verification enabled is 'yes', Claims and Level of confidence options is shown.

Add another item to a list

There is a new pattern included within the prototype. This enables us to add more than one item to a list of text inputs.

The user is given the option to add an additional item, which they are then given the option to remove.

There is no pagination on the page, regardless of how many email addresses have been added. All email address inputs should appear on one page.

This pattern can be seen on the following pages:

  • Enter contacts
  • Enter redirect URIs

Error scenarios and messages

Error messages to follow usual GDS pattern - error summary at the top, with error repeated above the field.
